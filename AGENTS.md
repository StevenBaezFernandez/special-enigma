Se implementa un Multi-Tenancy Híbrido de Alta Densidad con tres modos de operación definidos operacionalmente y automatizados en el provisioning. La selección de modo se basa en reglas de negocio + thresholds técnicos + requerimientos regulatorios (véase 4.4). Se exige telemetry y coste atribuido por tenant en todo momento. Modos operativos y ajustes (actualizados y con criterios cuantificados): 1. Shared Schema + RLS (High Density) Uso: micro-tenants, free tier, pruebas rápidas. Requisitos: políticas RLS obligatorias por tabla. Índices compuestos por (tenant_id, created_at, <otros campos selectivos>). Tests de performance: POC obligatorio que demuestre < 200 ms p95 para queries comunes bajo 1k QPS por tabla con 100k tenants sintéticos (ver sección 20 POC). Limitaciones: máximo recomendado: 100k filas por tenant en planes free; límites automáticos de cuota por tenant enforced por gating y throttling. La política de soft-limits: cuando un tenant supera el 75% del límite, se envía recomendación de migración; al 100% se bloquean writes hasta revisión/upgrade. Monitoreo: métricas por tenant de latencia de queries, cardinalidad de índices, contadores de filas; alertas automáticas cuando una tabla supere umbral por tenant (ej.: 75k, 90k, 100k filas). Reglas de acción automatizadas (recommendación/limitación/escala). 2. Schema-per-Tenant (Moderate Isolation) Uso: PyMEs, clientes con necesidad de separación lógica fuerte. Requisitos: orchestration de schemas (migraciones atómicas por schema). Naming convention tenant_<id>. Migration tool: migraciones idempotentes y dry-run obligatorio. Se exige prueba de restore a instancia de staging por tenant en el pipeline de onboarding. Backup/Restore: backups por schema con capacidad de restore a instancia de staging para auditoría. Restore validado por checksum y reconciliación de contadores. Securización: roles DB dedicados por schema, encryption keys rotables por schema, identificación de owners y políticas IAM. 3. Database-per-Tenant (Full Physical Isolation) Uso: Enterprise con requerimientos regulatorios o BYOK. Requisitos: aprovisionamiento automatizado (IaC) con KMS separado, posibilidad de llaves BYOK, backup/restore individual y SLA específico del tenant. Operación: métricas de coste por tenant, posibilidad de “cold storage” y snapshot retention extendida según contrato. Se exige contrato separado con SLAs financieros. Criterios de Selección Automática (Business Rules) Se automatizan decisiones de migración entre modos basadas en: consumo de recursos (CPU, IOPS, storage), requisitos regulatorios (data sovereignty), solicitud contractual y riesgo fiscal. El sistema de billing y governance puede iniciar “recomendaciones” de migración y ejecutar migraciones con aprobación manual por el equipo de onboard. Reglas numéricas de ejemplo (configurables): * Migrar a schema-per-tenant si: storage > 50GB o IOPS sostenido > 1000 por 7 días; o si tenant solicita explícitamente separación lógica. * Migrar a db-per-tenant si: tenant etiqueta regulatory:requires_physical_isolation o si coste de multi-tenant por tenant supera X$ mensual y cliente acepta coste. * Reversión automática deshabilitada; todo cambio de modo requiere aprobación manual y pre-migration checks. Contexto Inmutable por Request Cada request transportará un contexto inmutable y firmado por token:

{ tenantId, userId, role, permissions, region, currency, language, taxJurisdiction, complianceProfile, requestId, provenance }

provenance tracks original ingress point (edge node ID, timestamp). * Tokens cortos con refresh controlado por Keycloak/Auth provider. Tokens con claims mínimos y expiración corta (por defecto 15 minutos). Session refresh tokens rotados y con revocación. Validación / tests * Tests de performance RLS con dataset ≥ 1B rows y 1k–5k QPS totales agregados; p95 < 200 ms para queries indexadas (POC). * Tests de migración schema→db automatizados en staging con rollback 100% reproducible y validación por checksum por tabla. --- ## 1.2 Aplicaciones del Ecosistema Optimizado Tabla operativa con responsabilidades, SLAs y runbooks mínimos requeridos (actualizada con métricas y acciones): | Aplicación | Tecnología | Rol | SLA Técnicos | Región Primaria | Región Secundaria | Runbook mínimo | | ----------------------- | ------------------------------------- | ------------------------------------------------------------------- | -------------------------------------------------: | --------------- | ----------------- | ------------------------------------------------------------------------------------------------------- | | virteex-api-gateway | NestJS + Fastify + GraphQL Federation | Gateway unificado con BFF integrado y compresión (Brotli preferida) | 99.99% uptime; Arquitectura de Librerías ## 2.1 Principio Fundamental Toda la lógica de negocio debe vivir en libs/, no en apps/. Las aplicaciones son contenedores que orquestan librerías e inyectan configuración. libs/ está versionado por contratos semánticos; breaking changes requieren migración explícita y roadmap. Se define proceso de migración automatizado con scripts migrate-libs que aplica transforms, pruebas y notas de compatibilidad. Criterios obligatorios: * Documentar breaking changes en CHANGELOG.md y en Backstage con impacto por consumer. * Tests de compatibilidad automatizados: ejecutar nx affected:dep-graph y pruebas de contratos en PR. * libs/ publicables tienen pipeline que genera package + SBOM + signature. ## 2.2 Estructura Basada en Clean Architecture + DDD (detallada) Capas y responsabilidades (prohibiciones explícitas) * domain: entidades, value objects, agregados, reglas de negocio puras. No depende de infra ni frameworks. Ejecución: pure functions, no IO. Coverage mínimo 80% en domain libs (medido con herramientas de coverage). * application: casos de uso, orquestación de domain services, interfaces (ports). Puede usar DTOs pero no concretar infra. Tests de integración de application con mocks de ports y contract tests con colaboradores. * infrastructure: implementaciones concretas de repositorios, adaptadores externos, drivers DB, clients HTTP. No puede contener lógica de negocio. Debe contener adaptadores con tests de integración (Testcontainers). * presentation: adaptadores web, controllers, GraphQL resolvers; deben delegar a application. E2E mínimos para rutas públicas. * contracts: DTOs y interfaces compartidas. Cualquier cambio rompedor a contracts requiere versionado semántico cross-repo y migración asistida. Se impone compatibilidad hacia atrás por dos versiones activas. Contratos versionados con schema registry (Avro/Protobuf) y breaking-change policy. MikroORM vs alternativas (ajuste de 10/10) * Se adopta MikroORM por su Data Mapper y compatibilidad con patterns DDD. Implementación de POC productivo que incluya: migraciones atómicas, read/write splitting, y adaptadores para custom types (JSONB, ranges, composite PK). * Estrategia: usar MikroORM para la mayoría de las entidades; para consultas analíticas o queries muy optimizadas usar query builders o SQL raw en infrastructure. Documentar patrones para evitar N+1 y anti-patterns. * Repositorio de pruebas: medir latencias medianas y p95 en queries críticas con y sin ORM; establecer “escape hatch” documentada (helpers raw SQL) con pruebas de rendimiento ≥ 2x mejor que equivalente ORM para queries críticas. Convenciones de diseño * Entidades inmutables cuando sea posible; side effects controlados por application services. * Eventos de dominio modelados y versionados; esquema de eventos almacenado en events con contract schema registry (Avro/Protobuf) versionado para integración con Kafka. Documentar proceso de migración de eventos y consumer compatibility. ## 2.3 Convención de Nombres por Plataforma Prefijos obligatorios y sufijos de estabilidad/version: * api- (NestJS) * web- (Angular) * mobile- (Capacitor/Ionic) * desktop- (Tauri/Electron drivers) * shared- (DTOs, assets) * kernel- (core utilities, auth, telemetry) * domain- (dominios de negocio) * infra- (adapters concretos, infra as code helpers) * ml- (si aplica, modelos) Regla: el nombre debe contener v{major} si rompe compatibilidad mayor (ej. api-orders-v2). Contracts son la única capa transversal; cambios en contracts requieren migración controlada. Se implementa compatibilidad hacia atrás por 2 versiones activas y tests automáticos de compatibilidad. ## 3.3 Configuración ESLint para Boundaries Avanzada Reglas ESLint y Nx enforceadas que incluyen: * Bloqueo por etiquetas (region:latam no se importa en region:us sin wrapper). * Prohibición de imports relativos fuera del scope y prohibición de require dinámico en domain. * Herramientas de análisis en pre-commit y CI que fallan el build si se detectan ciclos o imports prohibidos. * Excepciones solo con ARCH_APPROVAL en PR y parámetros de justificación; aprobación caduca en 30 días si no se convierte en ticket de backlog. ## 3.4 Comunicación entre Dominios Patrón: Transactional Outbox Pattern con Debezium + Kafka (o poller idempotente) * Cada transacción de negocio escribe en la tabla outbox en la misma transacción. Los eventos llevan eventId, sourceService, sequenceNumber, schemaId, createdAt. * Un proceso asíncrono (Debezium o poller idempotente) publica en Kafka. Se exige exactly-once semantics a nivel de aplicación mediante idempotency keys y deduplicate in consumers. * Reconciliaciones automáticas: job nocturno compara outbox vs topic offsets y reintenta faltantes; alerting si diferencias > threshold (ej. >100 eventos o > 0.01% en 24h). * Eventos versionados y con contract schema registry (avro/protobuf) y tests end-to-end in Testcontainers. * Test de integridad: end-to-end tests que levantan DB + Debezium + Broker en Testcontainers, validan idempotencia, orden y esquema. ## 3.5 Herramientas de Detección de Ciclos * Dependency Cruiser en pre-commit y CI. * Job en CI que genera grafo de dependencias y diff con baseline; cualquier degradación falla la pipeline. * Política de deuda técnica: ciclos requieren ticket en backlog y aprobación de comité para merge urgente; las excepciones tienen tope temporal y métricas de deuda. --- # 4. Reglas de shard assignment * Basado en tenant attributes: taxJurisdiction, latency preferences, legal constraints. * Fallback geográfico configurable y visible en Backstage. * Cross-region replication solo vía procesos autorizados con enmascaramiento y controles (Data Sovereignty Guardian). Replication con masking para PII y autorización clara. * Monitoreo de residuo de datos y auditoría de replicaciones. ## 4.4 Operación, Migración y Costing por Tenant-Mode Cost attribution * Métricas: CPU, Memoria, Storage, IOPS, Snapshots por tenant. Deben recogerse en un bucket de métricas por tenant para FinOps. * Mostrar coste estimado en el portal de admin para clientes Enterprise y estimación en onboarding. * Automatización para migrar tenants según thresholds de coste o cumplimiento. Migración automatizada * Workflow: snapshot -> validate -> migrate schema/data -> switch routing -> monitor 24h -> finalize. * Rollback automático si errores críticos ocurren en verificación. * Tests de integridad inmediatamente después de migración (checksum por tabla). * Gates: migración bloqueada si test de performance en staging no pasa (p95/p99 thresholds). * Validaciones post-migración: reconciliation de counts, integrity hashes, y pruebas de flows críticos (timbrado, cobros). --- # 5. Seguridad y Compliance Completo (end-to-end) ## 5.1 Framework de Seguridad por Capas (detallado) Defensa en profundidad con responsabilidades claras: * Perimetral: WAF (AWS WAF o equivalente) con reglas personalizadas para OWASP Top 10 + rate limiting + geo-blocking configurable. Implementar "learning mode" previo a reglas bloqueantes en producción. * Identity & Access: Keycloak como IdP primario con HA, soporte SAML/OIDC; opción de Auth0 para enterprise con integración BYOID. SSO obligado para administradores; MFA obligatorio para todos usuarios con roles privilegiados. Bloqueo y listas negras por detección de anomalías. * Secret Management: HashiCorp Vault con auto-rotate de claves y integration con CI para env vars dinámicas; acceso mediante short-lived credentials. Secrets engine con policy RBAC granulada. * Network Segmentation: VPC, subnets por capa, security groups con least privilege. Zero trust dentro de VPC (micro-segmentation). * DDoS Protection: AWS Shield Advanced o proveedor equivalente con playbook de incident response y escalamiento. * Runtime Protection: EDR en hosts críticos, runtime detections y alertas a SOC. Instrumentación para runtime anomalies y honeypots de detección. * Data Protection: Tokenization/Field-level encryption para PII y campos fiscales; tokenization reversible en entorno controlado y auditado. Hardening concreto * Imágen base validated y signed; scanning de CVEs antes de publicación.
