
## 2. Motor Fiscal y Facturación (Revenue Core)

Este es el módulo más crítico. Actualmente contiene múltiples stubs y adaptadores "Null" o "Mock" que simulan éxito en lugar de realizar transacciones reales.

### 2.1. Proveedores de Certificación (PAC / DIAN)
- **FinkokPacProvider (MX):**
    - **Estado:** Implementado con lógica real de consumo SOAP (`axios` + `fast-xml-parser`).
    - **Problema:** El manejo de errores es básico. Si Finkok retorna un error 500 o timeout, el sistema simplemente reintenta 3 veces y falla. No hay un mecanismo de "Circuit Breaker" para evitar saturar el servicio externo.
    - **Mejora:** Implementar `opossum` o similar para circuit breaking. Persistir el XML firmado y el UUID *antes* de confirmar la transacción al usuario para evitar pérdida de datos en caso de fallo de red post-timbrado.
- **NullPacProvider (Fallback):**
    - **Estado:** Usado por defecto si no se configura el país o credenciales.
    - **Riesgo:** En producción, esto podría causar que facturas "timbradas" sean inválidas legalmente.
    - **Acción:** Deshabilitar `NullPacProvider` en entornos `production`. Forzar configuración de PAC válida al arrancar el servicio.
- **DianFiscalAdapter (CO):**
    - **Estado:** El código base (`libs/domains/fiscal/infrastructure/src/lib/adapters/dian-fiscal-provider.adapter.ts`) muestra un esqueleto vacío o básico. Falta la implementación completa del firmado XAdES-EPES con la llave privada del contribuyente y el envío SOAP al WS de la DIAN.
    - **Acción:** Desarrollar la lógica de firmado XML-DSig (Enveloped) y consumo del WSDL de la DIAN (Habilitación / Operación).

### 2.2. Cálculo de Impuestos (Tax Engine)
- **TaxRuleEngine:**
    - **Estado:** Funcional básico. Recupera reglas de BD (`TaxRule`).
    - **Problema:** La estrategia de impuestos de EE.UU. (`UsTaxStrategy`) usa una tasa fija (`DEFAULT_STATE_RATE = 0.0825`). Esto es incorrecto para un SaaS real en USA, donde las tasas varían por condado/ciudad (Zip Code).
    - **Mejora:** Integrar una API de cálculo de impuestos en tiempo real (Avalara, TaxJar, Vertex) para direcciones de EE.UU., o implementar una tabla de tasas por Zip Code mantenida internamente.
- **Race Conditions en Facturación:**
    - **Estado:** `CreateInvoiceUseCase` guarda la factura en estado `PENDING`, llama al PAC, y luego actualiza a `STAMPED`.
    - **Riesgo:** Si el servidor se reinicia o falla *durante* la llamada al PAC, la factura queda en `PENDING` (o el usuario reintenta y crea duplicados fiscales).
    - **Solución:** Implementar **Idempotencia** estricta usando `Locking Optimista` (versión en BD) o un `Distributed Lock` (Redis) por `invoiceId` durante el proceso de timbrado. Usar un `Background Job` (BullMQ) para reintentar transacciones en estado "desconocido" (consultar status en PAC antes de re-timbrar).

### 2.3. Generación de Documentos (XML / PDF)
- **Builders:** `MxFiscalDocumentBuilder` y `CoFiscalDocumentBuilder` existen pero parecen generar XMLs simplificados.
- **Validación:** No hay validación contra esquemas XSD oficiales (SAT v4.0, DIAN UBL 2.1) antes del envío. Esto delega la validación al PAC, lo que es costoso y lento.
- **Acción:** Integrar validación XSD local (`libxmljs` o similar) para rechazar payloads inválidos rápidamente y reducir costos de llamadas a API fallidas.
